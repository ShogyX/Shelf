SECURITY (cross-cutting — address first)

[HIGH] backend/app/epub_export.py:90 — SSRF in resolve_image_bytes()
  resolve_image_bytes fetches arbitrary remote http(s) chapter-image URLs with
  httpx.get and NO SSRF guard, unlike the rest of the codebase
  (imagecache.cache_image calls netguard.assert_public_url). A chapter that
  references http://169.254.169.254/... or an internal host gets fetched
  server-side during EPUB/Kindle export.
  FIX: route the fetch through netguard.assert_public_url before requesting.

[HIGH] backend/app/epub_export.py:120 — SSRF in _load_cover()
  Same gap as above: _load_cover fetches any http(s) cover_url directly via
  httpx.get with no public-address check.
  FIX: guard with netguard.assert_public_url.

[MEDIUM] backend/app/main.py:151-152 — /covers and /media served without auth
  /covers and /media are mounted as StaticFiles with no auth dependency, so any
  unauthenticated client can fetch cached chapter images / comic page images /
  cover art directly (e.g. /media/imgcache/<hash>.<ext>), bypassing the per-user
  library isolation enforced on every other route. Content is hash-addressed
  (not trivially guessable) but is served with no session.
  FIX: gate these mounts behind an auth-checked route, or move private media
  (comic pages) behind require_auth while leaving public covers open.

[MEDIUM] backend/app/kindle.py:124-129 — credentials sent over plaintext SMTP
  When cfg.starttls is false AND cfg.ssl is false, the code still calls
  server.login(username, password) over an unencrypted connection, sending
  credentials in cleartext. Nothing enforces an encrypted channel before AUTH.
  FIX: require STARTTLS/SSL (or refuse to send credentials) when a username is
  configured.

[LOW] backend/app/routers/auth.py:187-229 — sessions not revoked on pw change / demotion
  update_user revokes sessions only on deactivation. Changing a user's password
  or demoting them from admin leaves existing UserSession rows valid, so a stale
  or compromised session keeps full (or stale-admin) access after a forced
  password reset / demotion.
  FIX: also revoke sessions on password change and on role downgrade.

[HIGH] backend/app/routers/reading.py:153-170 — series auto-advance is dead code
  save_progress is a synchronous def route, so it runs in FastAPI's threadpool,
  which has no running event loop. asyncio.get_running_loop() at ~167 therefore
  ALWAYS raises RuntimeError, which is swallowed — so the series auto-advance
  background task never fires from the HTTP path. The feature is silently dead.
  FIX: make the route async def (and await/schedule the task on the live loop),
  or dispatch the background work via run_coroutine_threadsafe / a threadpool.

[MEDIUM] backend/app/routers/index.py:591-665, 1037-1170, 1258-1314 — sync DB in async routes
  Multiple async def catalog endpoints (list_catalog, grab_pipeline,
  acquire_catalog, hook_catalog, catalog_releases, etc.) perform many synchronous
  SQLAlchemy calls (db.get, db.scalars, commits) directly in the coroutine,
  blocking the single event loop for each DB round-trip and serializing all
  concurrent requests. Only the heavy grouping is offloaded via asyncio.to_thread.
  FIX: run the sync DB work in a thread (asyncio.to_thread) or make these sync
  routes so they run in the threadpool.

[LOW] backend/app/routers/delivery.py:230-237 — admin cannot bulk-download another user's shelf
  _bulk_zip rejects a shelf whose user_id != user.id with no admin bypass, which
  contradicts the admin-can-access-any-library intent used by _target_user_id /
  in_library in sibling endpoints.
  FIX: add a user.role == "admin" exception (if cross-user admin access is intended).

[LOW] backend/app/routers/goodreads.py:118-135 — uncaught 500 in test_my_goodreads
  test_my_goodreads only catches IntegrationError; meta_mod.provider_for(integ)
  or a non-IntegrationError raised by test_connection() propagates as an uncaught
  500 instead of the structured IntegrationTestOut(ok=False, ...) the endpoint

[HIGH] backend/app/ingestion/stock.py:264-281 — libgen download runs inside stock_tick
  _try_libgen calls libgen.fetch_for_stock (a full synchronous download+verify)
  from inside _process_pending, which runs inside stock_tick. The module docstring
  and stock_libgen_tick explicitly state libgen must NOT run in stock_tick because
  it blocks for minutes on slow mirrors — yet the pending path invokes it on every
  "no usenet release" / IntegrationError, defeating the decoupling and stalling
  reconcile/progress. (Directly contradicts the shared-SAB backpressure design.)
  FIX: route the fallback through the issue queue (mark unavailable / enqueue)
  instead of downloading inline in stock_tick.

[HIGH] backend/app/ingestion/libgen.py:167-171 — concurrency semaphore swapped while in use
  _semaphore mutates the module-global _concurrency whenever a config with a
  different max_concurrent is seen, replacing the live semaphore object. Concurrent
  ticks/grabs already holding the OLD semaphore no longer share the cap with new
  acquirers, so the global concurrency cap can be silently exceeded — defeating the
  purpose of the cap on a shared/abused endpoint.
  FIX: create the semaphore once and never swap it while in use (resize via a
  dedicated mechanism, or fix the cap at startup).

[HIGH] backend/app/ingestion/scheduler.py:248-253 — failure handler zeroes job.attempts mid-batch
  In _process_job's generic except Exception, on the 5th failure it sets
  ch.fetch_status = "failed" and resets job.attempts = 0 but does not continue/break;
  the loop proceeds to the next pending chapter with attempts just zeroed. job-global
  attempts conflates distinct chapters, so a work with many bad chapters never trips
  the cumulative-attempt guard and a transient burst keeps retrying the whole batch.
  FIX: track attempts per-chapter, or stop processing the rest of the batch after a
  chapter is marked failed, rather than zeroing the job-global counter.

[MEDIUM] backend/app/ingestion/scheduler.py:489-494 — reaper can double-run a live job
  The reaper revives a job stuck "running" past _STUCK_RUNNING_S based only on
  started_at. tick() selects jobs with status in (scheduled, running); after the
  reaper flips a job back to scheduled, the next tick can re-pick it even though the
  original _process_job coroutine may still be alive — letting two coroutines process
  the same work's pending chapters concurrently and double-fetch.
  FIX: gate revival on a liveness signal or a generation/lease token, not elapsed
  wall time alone.

[MEDIUM] backend/app/ingestion/scheduler.py:172-174 + 257-262 — periodic refresh cadence lost
  A refresh job (kind=="refresh") that keeps finding pending chapters falls through
  to the pending-fetch loop and is rescheduled as a backfill-style job;
  schedule_refresh_jobs won't create a new periodic refresh because an open job
  already exists. Net effect: the periodic ~6h refresh cadence is lost for any work
  whose refresh job keeps draining chapters.
  FIX: keep refresh jobs distinct from backfill rescheduling so the periodic cadence
  is preserved.

[MEDIUM] backend/app/ingestion/stock.py:255-261 — StockItem stranded in "downloading"
  reconcile_stock treats job is None as "skip". A StockItem can reference a libgen
  job that was left queued (transient retry) then given up / pruned by cleanup_jobs;
  the StockItem only reconciles on imported/failed, so a permanently-requeued-then-
  pruned libgen job strands the item in "downloading" forever with no completion path.
  FIX: reconcile StockItems whose referenced job is missing/pruned (treat as failed),
  and ensure the libgen worker propagates terminal state to the StockItem.

[MEDIUM] backend/app/ingestion/downloads.py:913-961 — piggyback follower orphaned / failed
  When the primary's nzo is in "queue", followers are grouped by their OWN nzo key;
  a follower whose nzo_id differs from the primary's (created before _repoint_followers)
  lands in a different group and is processed as a lone follower. The _live_primary
  re-home only triggers when not primary.candidates, so a stale follower past
  _STALE_AFTER can be FAILED at ~961 even though the real download is alive —
  orphaning the follower's library add.
  FIX: re-home/repoint followers to the live primary's nzo before staleness checks;
  don't fail a follower whose primary is still active.

[MEDIUM] backend/app/ingestion/tracker.py:103-105 — total_chapters_known oscillates
  _reseed_sequential flips a "skipped" (dead-end) frontier chapter back to "pending"
  and returns 1; discover_updates then counts total excluding only "skipped" rows, so
  the re-probed (now pending) row inflates total_chapters_known each refresh. Combined
  with _finalize_done reconciliation this can briefly show fetched > total or a
  regressing total.
  FIX: exclude speculative re-probe rows from total_chapters_known, or count them
  consistently across refreshes.

[LOW] backend/app/ingestion/libgen.py:299-325 — leftover .part temp files
  In Fetcher.download, on an exception after open(tmp,"wb") (e.g. OSError mid-write),
  the partial dest+".part" can be left behind; _cleanup only removes path and
  path+".part" for the candidate dest, so .part files from differently-named retries
  accumulate in the staging dir.
  FIX: clean up the temp file in a finally/except for the actual tmp path written.

[LOW] backend/app/ingestion/downloads.py:486 — non-atomic _promote race (libgen path)
  _promote does os.remove(dest) then shutil.move(src_file, dest) with no atomicity.
  The SAB poll path is serialized by _poll_lock, but the libgen path (_import_file ->
  _promote) is NOT under _poll_lock and can run concurrently with a SAB import for the
  same promoted path; one move can fail after the other removed/moved, spuriously
  failing a verified download.
  FIX: serialize promotion for a given dest, or make the move atomic and tolerate the
  already-moved case.

[LOW] backend/app/ingestion/fetcher.py:185 — advertised per-host daily cap not enforced
  SourceBudget.acquire never raises DailyBudgetExceeded; the indexer's budget-cooldown
  branch (kind="budget") is therefore dead code, and the per-host daily request cap
  described in configure_source is not enforced for crawl (only libgen has its own).
  FIX: either enforce the daily cap in acquire() or remove the dead config/branch and
  docstring claim.

[LOW] backend/app/ingestion/scheduler.py:54 — _seconds_until_window over-waits
  hours = (s - now.hour) % 24 or 24 can over-wait nearly a full day at an hour
  boundary when the window opens later the same hour. Minor resume-timing off-by-hour.
  FIX: compute the next window-open instant precisely from minutes, not whole hours.

[MEDIUM] backend/app/integrations/ratelimit.py:38-42 — lock held across sleep serializes callers
  throttle holds the per-key asyncio.Lock across await asyncio.sleep(wait), so all
  callers for a key serialize on the sleep itself. With a low rpm (e.g. NovelUpdates
  rpm=6) and a burst of queued enrich/release tasks, callers block cumulatively for
  many minutes with no timeout/cancellation budget.
  FIX: reserve the next-slot timestamp under the lock, then perform the sleep OUTSIDE
  the lock.

[LOW] backend/app/integrations/sabnzbd.py:156 — history() not paginated

  the queue page-window and the history window is eventually failed as "SABnzbd no
  longer tracks this download" (the _STALE_AFTER grace makes this low-probability).
  FIX: add a paginated history_all (mirror queue_all).

[HIGH] backend/app/db.py:441 — per-user reading_states unique-index backfill can crash
  _migrate_reading_states_per_user creates a UNIQUE index on (user_id, work_id), but
  legacy rows get user_id backfilled to the first admin. SQLite treats multiple NULL
  user_ids as distinct, so a later backfill assigning the SAME admin id to two
  NULL-user rows of the same work violates the unique index and crashes the backfill.
  FIX: de-dup (coalesce/merge) reading_states per (work_id) before assigning the same
  owner, or assign distinct ownership.

[HIGH] backend/app/backup.py:104,110 — NULL natural keys break restore dedupe
  _NATURAL_KEY["reading_states"] = ("user_id","work_id") (and chapters by
  ("work_id","index")) is used to dedupe on merge. For reading_states, legacy NULL
  user_id rows make the key tuple contain NULL; SQLite =/IN never match NULL, so on
  merge these rows are inserted as duplicates and can violate the UNIQUE index,
  aborting the whole atomic restore.
  FIX: coalesce NULLs (or skip natural-key dedupe when a key column is NULL).

[MEDIUM] backend/app/routers/works.py:430 — catalog cache not cleared on removal/purge
  remove_from_library / _prune_if_orphaned (which purges the Work + catalog rows) does

  showing the removed/purged title as in_library/in_stock for up to the 120s TTL and
  can reference a now-deleted work.
  FIX: clear the catalog cache on removal/orphan-purge (as the hook path does).

[MEDIUM] backend/alembic/versions/0020_catalog_discovery.py:88 — catalog_categories unique key mismatch

[MEDIUM] backend/alembic/versions/0020_catalog_discovery.py:48 — missing is_adult columns
  catalog_groups is created without is_adult, and there is no migration adding

[MEDIUM] backend/app/imagecache.py:119,157 — concurrent cache_image leaves stale .fail marker
  cache_image checks _existing_local (fixed ext list) then writes {name}.{ext} with ext
  from the response content-type; the fail-marker/file writes are non-atomic. Two
  concurrent crawler threads can both miss _existing_local, both fetch, and one can write
  a .fail marker while another writes the real image under a different ext — leaving both
  a .fail and a real file. Subsequent calls hit the .fail first and return PERMANENT_FAIL
  even though a good copy exists.
  FIX: check for an existing real image before honoring the .fail marker (and/or write
  atomically; skip the marker when a real file exists).

[LOW] backend/app/routers/index.py:665 — /catalog browse always reports in_library=False

[LOW] backend/app/db.py:298 — _ADDITIVE_COLUMNS dict silently drops duplicate table keys

[CRITICAL] backend/app/backup.py:~657 — Zip-slip / path traversal → arbitrary file write (RCE)
  _restore_media computes dest = root / name[len("media/"):] directly from the zip
  entry name with NO containment check, then mkdir(parents=True) + writes. A crafted
  backup zip with an entry like media/../../../../etc/cron.d/x (or an absolute path)
  writes outside media_dir. Restore is admin-gated and admins upload arbitrary zips
  via POST /admin/backups/upload (only the manifest is validated, not entry paths),
  so a tricked/compromised admin or a malicious "shared" backup achieves host file
  write → RCE.
  FIX: resolve dest and assert dest.resolve().is_relative_to(root.resolve()); reject
  any entry that is absolute or contains "..". Apply the same check to every zip
  entry consumed during restore.

[HIGH] backend/app/auth.py:131-163 — login brute-force throttle is per-process in-memory
  _fail_log is an in-memory dict guarded by a thread lock. Under any multi-worker
  deployment (gunicorn/uvicorn --workers N) the per-username/per-IP counters are not
  shared, so the effective attempt cap is multiplied by worker count and resets on
  restart — materially weakening brute-force protection. (Same single-process
  assumption affects the libgen semaphore and fetcher rate caps — see design issues.)
  FIX: back the counter with the DB (a login_attempts table) or a shared store, or
  hard-document/enforce single-worker only.

[MEDIUM] backend/app/epub_export.py:90,120 — SSRF in export image fetch
  (Cross-ref Part 1 HIGH x2.) Confirmed again here, with the added note that
  follow_redirects=True lets a public URL 302 to 169.254.169.254 / RFC-1918.
  FIX: assert_public_url before each fetch AND disable redirect following (or
  re-validate each hop).

[LOW] backend/app/ingestion/netguard.py:55-65 — SSRF guard is TOCTOU / DNS-rebinding-susceptible
  Resolves host via getaddrinfo and validates those IPs, but httpx/socket re-resolves
  independently at connect time; a low-TTL hostname can pass the guard then connect to
  an internal IP.
  FIX: resolve once and pin the validated IP for the actual connection, or re-validate
  the connected peer address.

[LOW] backend/app/routers/auth.py:37-48,92-99 — first-run setup IP gate fails OPEN behind untrusted proxy
  When no setup_token is configured, setup is refused only if client_ip is "definitely
  public." If exposed without trust_proxy, request.client.host is the proxy/loopback
  (private) address, so _is_public_ip returns False and the tokenless first-admin claim
  is allowed to any internet client racing to /auth/setup.
  FIX: require the setup token whenever the deployment is internet-reachable, or fail
  closed when proxy trust isn't configured.

[LOW] backend/app/config.py:58 — cookie_secure defaults False, no CSRF token
  On plain-HTTP the session cookie is sent in cleartext; CSRF protection relies solely
  on SameSite=lax (which does block cross-site form POSTs). _is_secure auto-upgrades
  behind a trusted https proxy.
  FIX: default Secure on / add an Origin check for state-changing requests.

[LOW] backend/app/sanitize.py:73-89 — <img src> scheme not validated
  Only <a href> blocks javascript:/data:/vbscript:. img src can't run script in modern
  browsers and CSP (img-src 'self' https: data:) neutralizes most of it, but data:-image
  exfil/tracking is possible if CSP is ever disabled.
  FIX: scheme-filter img/src too (allow only http(s)/relative//media).

[LOW] backend/app/epub_export.py:29-32,151 — /media path mapping lacks traversal containment
  _local_image_bytes maps /media/<...> to media_dir()/<...> with no containment, while
  /covers/ is safely basename-ed. Not currently reachable with attacker-controlled
  traversal (src is sanitized/hashed) but the asymmetry is fragile.
  FIX: confine the resolved path under media_dir() like the covers branch.

================================================================================
DEEP CONCURRENCY / RACE CONDITIONS (cross-module)
================================================================================
Dominant theme: lifecycle state on shared DB rows (CrawlJob, StockItem, DownloadJob,
WatchedFolder, Work.local_path) is coordinated by SELECT-then-act guards across
independent sessions with NO unique constraints or row locks. Every "another run may
have created one concurrently" comment marks a window the comment does NOT actually
close. The structural fix for most HIGHs below: add unique constraints
(StockItem.norm_key, Work.local_path per source, partial-unique active CrawlJob(work_id))
+ IntegrityError-reuse, and give the download/import path one canonical writer.

[CRITICAL] scheduler.py:455-563 (reap_stalled_jobs) vs tick()/_process_job — two writers on CrawlJob
  The reaper (APScheduler threadpool) flips a job it deems stuck (running > 600s) to
  scheduled/now while _process_job (event-loop thread) is still committing status/cursor/
  scheduled_for to that same job from a different session. No row lock or version check →
  lost updates and a live job revived as a duplicate, re-picked by the next tick (the
  seen_works guard only dedups within one tick, not across the reaper). (Part 1 flagged
  the revival-of-live-job symptom; this is the underlying two-writer data race.)
  FIX: single-writer discipline for job lifecycle (only reaper OR only tick mutates), or
  optimistic version column / re-read started_at under a lock before reviving.

[HIGH] downloads.py:833-969 (poll_tick) vs grab_release — duplicate primary DownloadJob
  poll_tick is NOT held under _grab_lock while a concurrent request-path grab_release IS.
  A grab_release for the same book can pass its dedup check (357-366) in the window
  between poll's _grab_next enqueuing a new nzo and committing, so both enqueue the same
  release → two DownloadJob primaries for one book.
  FIX: hold _grab_lock across poll's primary-advance group handling too, and/or add a
  partial unique index on (catalog_work_id, status active).

[HIGH] stock.py — StockItem marked "stocked" by three uncoordinated paths
  (1) _import_completed → on_stock_imported (poll), (2) reconcile_stock → _mark_stocked
  (run at top of both stock_tick and stock_libgen_tick), (3) _process_pending inline.
  These run on the same loop with separate sessions and interleave at awaits; a stale
  _mark_stocked commit can overwrite, and _migrate_work_links can run TWICE for the same
  old_id→new_id, double-deleting/moving LibraryItem/Chapter rows.
  FIX: make _mark_stocked/on_stock_imported idempotent (early-return when
  si.status=='stocked' and si.work_id==work_id), and serialize the stock+libgen+poll
  import handlers behind one lock/worker.

[HIGH] downloads.py:232-257 (ensure_watched_folder) + watcher.py — import races delete promoted file
  _import_completed promotes a file then creates the Work; concurrently the watchdog
  observer thread and the periodic folder rescan can sync_folder the same just-promoted
  file with independent sessions. Ordering interleavings can make _import_completed's
  Work lookup find none and fall into the orphan-removal branch (587) that DELETES the
  just-promoted file out from under the in-progress sync, or create duplicate Works.
  FIX: unique constraint on Work.local_path (INSERT-OR-IGNORE semantics) and don't delete
  a promoted file when a concurrent sync may own it.

[HIGH] downloads.py:473-491 (_promote) — non-atomic overwrite, no per-destination lock
  (Cross-ref Part 1 LOW.) Confirmed higher severity: SAB and libgen paths can promote a
  verified file for the SAME book into the same lib_dir/<title>/<basename> concurrently;
  the promoted basename is os.path.basename(src) (NOT job-scoped), so collisions are real.
  A removes dest, B removes (no-op), A moves, B moves → clobber or mid-move failure.
  FIX: promote to a temp name + atomic os.replace, or lock per destination dir.

[HIGH] libgen.py:162-189 — concurrency semaphore swapped mid-flight; blocked_until set outside lock
  (Cross-ref Part 1 HIGH on semaphore swap.) Additionally: _HostState.blocked_until in
  _note_backoff (192-194) is written OUTSIDE st.lock, so a backoff can be lost under
  concurrent ticks.
  FIX: build the semaphore once at config load and pass it through; set blocked_until
  under st.lock.

[MEDIUM] stock.py:135-174 (queue_selection) — duplicate StockItem for one norm_key
  Two concurrent admin POSTs both pass the per-row SELECT(StockItem.id where norm_key==nk)
  guard and both insert; there is NO unique constraint on StockItem.norm_key, so the worker
  grabs/downloads the same book twice.
  FIX: add a unique constraint/index on StockItem.norm_key and catch IntegrityError.

[MEDIUM] integrations/sync.py:36-59 (_map_one_folder) — unhandled IntegrityError aborts sync batch
  _map_one_folder does an exists-check then add+commit with NO IntegrityError handling,
  unlike ensure_watched_folder which catches+reuses. Two syncs (or a sync racing an import)
  for the same path → the second commit raises IntegrityError, rolling back the whole
  integration sync batch.
  FIX: wrap the commit in the same IntegrityError-reuse pattern as ensure_watched_folder.

[MEDIUM] scheduler.py:455-563 — reaper-vs-refresh/hook creates duplicate open jobs per work
  The reaper's re-check (541) won't see an uncommitted insert from schedule_refresh_jobs

[MEDIUM] cache.py:28-53 — clear-vs-put TTL race serves stale snapshot

[LOW] scheduler.py:1142-1145 / 1256-1261 — _scheduler global lifecycle unsynchronized

================================================================================
ANTI-BOT / CLOUDFLARE CHALLENGE HANDLING
================================================================================
Architecture is sound in intent (central RateLimited, adaptive throttle, job cooldowns,
render fallback) but challenge DETECTION has real holes: all marker scans cap at the
first 4 KB, RenderedPage exposes no headers, the browser pre-emptively rewrites
403/503→200, HTTP-200 challenges are never body-scanned, and store_chapter_content has
no independent backstop. Net risk: a verbose "Just a moment" page gets saved as a chapter
and checksummed, permanently poisoning the work.

[HIGH] backend/app/ingestion/browser.py:155-164 — "challenge cleared" heuristic scans only html[:4000]
  Cloudflare markers (and the real <title>Just a moment…</title>) often sit past 4 KB on
  large templated pages; a still-challenged 403/503 whose markers fall past byte 4000 gets
  rewritten to status=200 and the interstitial HTML is returned as real content, then
  parsed and stored.
  FIX: scan the WHOLE rendered HTML (and page.title()) for challenge markers before
  downgrading status to 200.

[HIGH] backend/app/ingestion/browser.py:117-132 — passive-challenge wait declares "clear" with no positive content signal
  When no wait_selector is given, the loop breaks the instant markers are absent from
  title+body[:4000], with no confirmation of real content; the marker list here is
  narrower than the fetcher's _BLOCK_MARKERS (no "attention required", "captcha-delivery",
  "turnstile", no cf-mitigated header check).
  FIX: unify on one shared marker set, check cf-mitigated/server headers, and require a
  minimum content signal before treating the page as cleared.

[HIGH] backend/app/ingestion/fetcher.py:505 — _looks_blocked passed headers that don't exist on RenderedPage
  RenderedPage (browser.py:35-40) has NO headers attribute, so getattr(page,"headers",{})
  is always {} and the render path can never see cf-mitigated. Combined with the 403/503→200
  downgrade, a missed challenge arrives with status=200 + empty headers → not blocked →
  challenge body returned as content.
  FIX: have RenderedPage carry response headers and the ORIGINAL (pre-downgrade) status;
  have _looks_blocked scan the full body, not getattr(page,"text","")[:4096].

[HIGH] backend/app/ingestion/engine.py:236-266 / scheduler.py:175-216 — store_chapter_content has no challenge backstop
  The only guard against persisting a challenge page is _is_dead_end (<50 words AND no
  <img). Cloudflare's verbose managed-challenge / "Attention Required" pages exceed 50
  words, so they get stored as a real chapter and checksummed — permanently poisoning the
  work.
  FIX: add a content-level challenge sniff in store_chapter_content (shared marker set vs
  sanitized body/title) that refuses to persist and raises RateLimited, independent of the
  fetcher.

[MEDIUM] backend/app/ingestion/fetcher.py:407 / 73-89 — HTTP-200 Cloudflare challenges never body-scanned
  _looks_blocked only inspects the body for 403/503; 200/3xx are treated as never-blocked.
  Cloudflare frequently serves managed-challenge/Turnstile interstitials with HTTP 200.
  FIX: scan the body on 200 HTML responses when cf-mitigated/server=cloudflare is present
  or the body is suspiciously small; check cf-mitigated regardless of status.

[MEDIUM] backend/app/ingestion/fetcher.py:387-392,413-414 — JS-only 403 block misclassified as permanent
  A genuine CF block can be a 403 whose markers live only in a JS-loaded body a plain HTTP
  client can't see (empty <noscript> shell); it's penalized and returned as "members-only"
  and never escalated to browser render.
  FIX: when a render-eligible source returns 403/503 with HTML content-type and a tiny/
  JS-only body, escalate to a browser render before classifying as forbidden.

[MEDIUM] backend/app/ingestion/fetcher.py — no per-host cooldown that survives blocks across requests
  penalize() sets _next_allowed_ts only when Retry-After is present; CF blocks usually send
  none, so the next request to the same host is gated only by interval×throttle, not a real
  cooldown. Job-level (scheduler) and site-level (indexer) cooldowns exist but a different
  code path hitting the same host shares none.
  FIX: on a detected block set _next_allowed_ts to a substantial, escalating host-level
  cooldown inside penalize so every caller of that host bucket waits.

[MEDIUM] backend/app/ingestion/browser.py:80-89 / fetcher.py:200-221 — no cf_clearance reuse between browser and httpx (engine path)
  The browser context's clearance cookies are in-memory only and are NOT shared with the
  plain-HTTP httpx client (unlike libgen.py:267-270 which does capture them), and are lost
  on restart — forcing a fresh challenge each time.
  FIX: persist clearance cookies (pin the exact UA that earned them) and inject into the
  httpx client for that host; consider on-disk storage_state so clearance survives restart.

[MEDIUM] backend/app/ingestion/browser.py:48-89 — inconsistent user-agents invalidate clearance
  Engine browser UA (settings, Mozilla/5.0 fallback fetcher.py:235), libgen hard-codes a
  different Windows Chrome 124 UA, browser fallback default is Linux Chrome 120. CF ties
  cf_clearance to the exact UA; mismatches re-trigger the challenge. The stealth init also
  advertises Linux while libgen claims Windows.
  FIX: derive one UA per fetcher/browser, use it consistently for render + cookie-bearing
  download, and make stealth navigator props match the declared platform.

[MEDIUM] backend/app/ingestion/libgen.py:201-206,313 — _is_cf_challenge scans only body[:4000], exact cf-mitigated match
  A JS-challenge whose markers sit past 4 KB is mislabeled "throttled" and retried via plain
  HTTP forever (never escalated to download_via_browser); cf-mitigated values other than
  exactly "challenge" (block, managed_challenge) are missed.
  FIX: scan the full body; treat any non-empty cf-mitigated as a challenge worth a browser
  attempt.

[LOW] backend/app/ingestion/browser.py:251-296 — capture_canvas_pages has no challenge detection
  If the reader URL returns a CF interstitial, the hydrate loop times out, returns (0,{}),
  and descramble.py:213-215 treats "no captured pages" as "already correct," marking the
  chapter checked — a block masquerades as a clean no-op.
  FIX: detect challenge markers / non-200 nav status in the capture path and surface a
  distinct signal so the descramble job retries.

[LOW] backend/app/ingestion/fetcher.py:73-89 — _BLOCK_MARKERS substring match false-positives / English-only
  A legitimate article quoting "captcha"/"access denied" trips a spurious RateLimited +
  cooldown; localized challenge text is missed.
  FIX: anchor markers to CF structures (<title>Just a moment, id="challenge-form",
  window._cf_chl_opt, cf-error-details) and/or gate the scan on a server: cloudflare / cf-ray
  header.

[LOW] backend/app/ingestion/fetcher.py:37-45 — _parse_retry_after drops the HTTP-date form
  Returns None for the IMF-fixdate form CF 429/503 commonly sends, so cooldowns can be much
  shorter than requested.
  FIX: parse the HTTP-date form and convert to a delta.
