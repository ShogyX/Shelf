================================================================================
SHELF — FIX IMPLEMENTATION PLAN
================================================================================
Companion to fix_issues.txt. That file is the catalog (WHAT + WHERE + WHY).
This file is the build plan (HOW + ORDER + shared infrastructure).

Read this top-to-bottom. Section 0 ("Foundations") must land FIRST: each
foundation is a small shared change that structurally resolves a whole cluster
of downstream findings. Doing the foundations first means most individual fixes
become one- or two-line call-site changes instead of bespoke patches.

Conventions used below:
  • "→ resolves" lists the fix_issues.txt items a change closes or de-risks.
  • Every behavioral fix needs a regression test (the suite is green today and
    asserts the buggy behavior — see fix_issues.txt Part 3). Test guidance is
    inline per workstream.
  • Touch DB schema only via a new alembic revision AND the boot-time
    _ensure_columns/_ensure_indexes path (the app runs both — see F1).

F0.1  SSRF guard helper — one outbound-fetch chokepoint
  WHERE: backend/app/ingestion/netguard.py (add a helper), then call sites in
         backend/app/epub_export.py (resolve_image_bytes ~90, _load_cover ~120),
         and audit every other httpx.get/Client outside imagecache/fetcher.
  HOW:  Add `def safe_get(url, **kw) -> httpx.Response` that (a) calls
        assert_public_url(url), (b) forces follow_redirects=False, (c) on a 3xx
        re-validates the Location and re-fetches (bounded hops), (d) sets a
        timeout. Replace raw httpx.get in epub_export with safe_get. ⚠ It MUST
        manually re-follow redirects (re-validating each hop), copying the proven
        loop in fetcher.py (~366-402): epub_export currently uses
        follow_redirects=True and legitimate cover/image CDNs 302, so merely setting
        follow_redirects=False and bailing on a 3xx (the imgproxy pattern) would
        BREAK those images. Factor out the fetcher loop so there is ONE definition.
  → resolves: epub_export SSRF (P0), and prevents the class from reappearing.
  TEST: a chapter/cover URL of http://169.254.169.254/ and a public→RFC1918
        redirect must raise, not fetch.

F0.2  Schema hardening: unique constraints + IntegrityError-reuse (kills the race cluster)
  WHERE: backend/app/models.py (constraints), a new alembic revision +
         backend/app/db.py _ensure_indexes (mirror), and the insert sites.
  HOW:  Add these constraints, then make every insert "upsert-or-reuse". The exact
        keys below are CORRECTED per the regression review (Section 12) — the naive
        versions break real behavior, so use these:
          - StockItem.norm_key UNIQUE — already declared in the model
            (uq_stock_norm_key) but NOT enforced on a live DB (create_all/
            _ensure_columns never retrofit constraints). Needs an alembic BATCH
            rebuild + dedupe existing dups first. norm_key is non-null and blank
            keys are already made unique (stock.py ~154 `nk = ... or f"id:{g.id}"`).
          - Work.local_path PARTIAL UNIQUE WHERE local_path IS NOT NULL (web works
            leave it NULL — must stay exempt). ⚠ Two WatchedFolders covering the
            same file legitimately create two Works with the same local_path; you
            MUST reconcile/dedupe those (and ideally scope the index by the local
            source) before enforcing, or dual-folder mapping breaks.
          - WatchedFolder.path UNIQUE — SAFE; already declared and ensure_watched_
            folder already does the IntegrityError-reuse. Use it as the template.
          - CrawlJob: PARTIAL UNIQUE on (work_id, KIND) WHERE status IN
            ('scheduled','running','paused'). ⚠ NOT (work_id) alone — a work
            legitimately has backfill + descramble (+ refresh) open at once, and
            _process_descramble_job depends on a concurrent open backfill
            (_has_open_backfill). Keying on work_id alone BREAKS the descramble
            pipeline and contradicts test_reaper.py. Must include kind AND 'paused'.
          - DownloadJob: ⚠ DO NOT ADD this constraint. Piggyback followers (the
            multi-user model) legitimately share catalog_work_id, and real dedup is
            per (norm_key CLUSTER, user_id), not per catalog_work_id. A unique on
            catalog_work_id breaks followers immediately. Keep dedup in code under
            _grab_lock (see C2) — no DB constraint here.
        Introduce ONE helper used at every NON-follower insert site:
          try: db.add(x); db.flush()
          except IntegrityError: db.rollback(); x = db.scalar(select(...existing))
        Apply at: stock.py queue_selection (~155, currently no IntegrityError
        catch), integrations/sync.py _map_one_folder (~36-59, no handler today),
        ensure_watched_folder (already does this — the template). NOT the
        DownloadJob follower path.
  → resolves: duplicate StockItem, duplicate WatchedFolder (sync batch abort),
        duplicate open CrawlJobs of the SAME kind, AND the reaper-vs-refresh
        duplicate-job window. (DownloadJob dup-primary stays a code/lock fix, C2.)
  NOTE: every constraint needs an alembic BATCH (table-rebuild) migration that
        FIRST dedupes pre-existing rows — plain create_all/_ensure_columns will NOT
        retrofit a constraint onto an existing table, and the rebuild fails if dups
        remain (same precedent as reading_states, F4.1).

F0.3  Single writer for job lifecycle (lease/generation token)
  WHERE: backend/app/ingestion/scheduler.py — tick()/_process_job vs
         reap_stalled_jobs (~455-563) and the _scheduler globals.
  HOW:  Add CrawlJob.lease_token (uuid) + CrawlJob.lease_expires_at. _process_job
        stamps a fresh lease on pick-up and only commits status/cursor if its
        lease still matches (optimistic check in the UPDATE WHERE). The reaper
        revives a "running" job ONLY when lease_expires_at < now (not raw
        started_at), and bumps the token so the abandoned coroutine's later
        commit is a no-op. This is the structural fix for the two-writer data
        race; combined with F0.2's partial-unique it also stops duplicate revival.
  → resolves: reaper-vs-_process_job CrawlJob race (P0/CRITICAL), reaper reviving
        a live job, lost job updates.
  TEST: simulate a "stuck" running job whose coroutine is still alive; assert the
        reaper's revive doesn't produce a second runner and the stale commit drops.

F0.4  Shared anti-bot challenge detector + RenderedPage headers
  WHERE: new backend/app/ingestion/challenge.py (or add to netguard); consumers
         browser.py, fetcher.py, libgen.py, engine.store_chapter_content.
  HOW:  (a) One marker set + one function
          `is_challenge(status:int, headers:Mapping, body:str) -> bool`
          that scans the FULL body (not [:4000]) and the <title>, treats ANY
          non-empty cf-mitigated header as a challenge, and recognizes 200-status
          challenges. Anchor markers to CF structures (<title>Just a moment,
          id="challenge-form", window._cf_chl_opt, cf-error-details) and gate on
          server: cloudflare / cf-ray to cut false positives.
        (b) Give RenderedPage (browser.py:35-40) a `headers` field AND carry the
          ORIGINAL pre-downgrade status as original_status. ⚠ DO NOT remove the
          403/503→200 downgrade (browser.py:160-164): it is correctly marker-gated
          (only downgrades when the challenge actually cleared and real content is
          present), and fetcher._render classifies on status — removing it makes
          every successfully-cleared render look "blocked" and regresses working
          render sources. Instead, ADD headers/original_status so _looks_blocked
          (fetcher.py:505) can judge, and have _looks_blocked use is_challenge on
          the full body.
        (c) Backstop in engine.store_chapter_content (~236-266): ⚠ do NOT free-text
          match challenge phrases against prose — real fiction contains "just a
          moment"/"captcha"/"enable javascript" and would be falsely rejected,
          losing content. Only reject when challenge markers CO-OCCUR with anti-bot
          signals (cf-* tokens, __cf_chl, challenge-platform) AND the page is short/
          imageless. Prefer tightening the existing _is_dead_end check (which already
          drops near-empty placeholder pages) over a new free-text scan.
        (d) libgen._is_cf_challenge: replace with is_challenge; treat non-empty
          cf-mitigated values (block/managed_challenge) as browser-worthy.
  → resolves: the entire Cloudflare cluster (Part 2): 4 HIGHs + the 200-challenge,
        JS-only-403, capture_canvas_pages no-op, and the "interstitial saved as a
        chapter" poisoning path.
  TEST: feed a verbose >50-word "Just a moment" body to store_chapter_content and
        assert it raises + is NOT persisted; feed a 200 + cf-mitigated to fetcher.

F0.6  Scheduler task wrapper + central logging + readiness
  WHERE: backend/app/ingestion/scheduler.py (~770-994), backend/app/main.py:37,

  HOW:  (a) `@scheduled_task(to_thread=False)` decorator that owns

        (b) Logging: honor SHELF_LOG_LEVEL env, configure once, set uvicorn

[HIGH] Soft-vs-hard block classification + escalating cooldown — fetcher.py:73-89,134-146,387-417 —

[HIGH] Auto-escalate plain-HTTP blocks to browser render — fetcher.py:407-409,419-444 — only sources

[HIGH] Persist + reuse cf_clearance across requests and restarts — browser.py:80-91,218-223,
  fetcher.py:200-221, libgen.py:269-272 — clearance the browser earns isn't shared into the httpx client
  and is lost on restart, so every restart re-solves every challenge. HOW: persist Playwright storage_state

[HIGH] SAB downloads stuck with no progress — downloads.py:913-959, sabnzbd.py:121-136 — a queued job is

  piggyback group) for 12h. HOW: record (percentage, mb_left, last_progress_at) each poll; if no byte

[HIGH] Per-host fairness + circuit breaker — fetcher.py:169,376 — one global semaphore (default 2) gates
  ALL sources; a slow/blocked host can hold both slots and starve everything, and a just-hard-blocked host is
  still dequeued and tried. HOW: add a per-rate_key semaphore (1-2) alongside the global cap, plus a

[MEDIUM] Per-source timeouts + stall-vs-slow distinction — fetcher.py:219,376-378, libgen.py:237,299-321 —

[MEDIUM] Detect no-forward-progress crawls (cursor stuck) — scheduler.py:212,217-233,489-503 — the reaper

[MEDIUM] Honor HTTP-date Retry-After + per-host realistic UA pool — fetcher.py:37-45,209-213,

[MEDIUM] Prioritize active-reading over backfill — scheduler.py:580-604 — tick() orders due jobs purely by

[MEDIUM] Health-rank libgen mirrors (dead-mirror demotion) — libgen.py:528-545,731-744 — hosts are tried in

[MEDIUM] libgen partial-download resume — libgen.py:299-309 — download() always restarts from byte 0; a

  is NO final tiebreak on e.id, so when members tie (common: both popularity 0, both have
  cover+synopsis — typical un-enriched rows) max() returns the first in scan order. A later
  enrichment bumping one member's popularity, or any insert/delete shifting the unordered
  `select(CatalogWork)` scan (catalog_groups.py:171), FLIPS the rep → group_id is rewritten →
  the React key changes and the work re-renders as a "new"/second card, and the 120s
  catalog-rows cache still holds the old id. HOW: derive the group id from min(member_ids)
  (earliest-discovered row — invariant under enrichment and scan order) and use rep selection
  only for DISPLAY fields; make every rep/_rep_key a TOTAL order with -e.id as final tiebreak.
  EFFECT: groups stop churning ids between the 10-min regroups → eliminates the transient
  double-render and stale-cache duplicates.

[HIGH] No external-ID identity resolution — catalog.py:352 _union_find_groups, models.py:333 —
  clustering is norm_key + token-Jaccard only, so romaji-vs-English ("Shingeki no Kyojin" vs
  "Attack on Titan"), native-only titles, and subtitle-only-on-one-source never merge, even
  though rows carry comix hid, provider_ref, and matchmeta already fetched AniList ids/synonyms
  into extra["alt_titles"]. HOW (uses K1): BEFORE the title union-find, union() any two rows
  sharing a non-empty identity_key (keep the media-bucket guard); also fold extra["alt_titles"]
  into the token postings so romaji↔English is caught by the existing fuzzy branch. EFFECT:
  merges the largest under-merge source (cross-source/cross-language variants, esp. comics)
  without loosening the title threshold.

[HIGH] CJK / empty-key over-merge AND single-token under-merge — extract.py:868-888, catalog.py:378,394,979 —
  norm_title strips all non-[a-z0-9] so every CJK/Cyrillic title → "" → in the exact-key bucket
  EVERY empty-key work in a media bucket collapses into ONE giant bogus group (catastrophic
  over-merge); separately, 1-token titles are excluded from fuzzy merge so identical one-word
  titles ("Bleach","Noragami") from two sources never merge (under-merge). HOW: (E1) keep CJK
  codepoints in norm_title; GUARD the exact-key bucket to NEVER union two rows when key=="" (treat
  empty key as ungroupable); allow exact-norm_key equality merge for 1-token titles (the exact
  pass already does this — only the empty-key collapse is the bug). EFFECT: removes the bogus
  mega-card and lets identical single-token/CJK titles group.

[MEDIUM] Late enrichment doesn't re-cluster; canonical-title corrections are inert — catalog_groups.py:151
  _should_regroup, scheduler.py:1206 — regroup is gated on a watermark of max(enriched_at,updated_at);
  enrichment bumps enriched_at but NEVER writes a corrected canonical title/norm_key/identity_key, so
  a provider's better title never re-clusters anything, and new merges wait ≤10 min. HOW: when
  enrichment/matchmeta learns a canonical title or external id, write it to norm_key/identity_key (so
  the watermark changes AND union-find actually re-clusters), and fire a debounced out-of-band
  regroup_catalog after an integration sync / hook. EFFECT: corrections actually collapse duplicates;
  shorter stale-duplicate window.

[MEDIUM] Fuzzy blocking skips high-degree tokens, dropping merges — catalog.py:397-400 (_MAX_BUCKET=200) —
  any token whose posting list >200 is skipped; for 2-token titles built from common words
  ("Dragon Ball","Solo Leveling") BOTH tokens can exceed the cap in a comic-heavy catalog → the pair
  is never compared → separate cards. HOW: block on each row's RAREST (min-posting) token instead of
  skipping large buckets, so every row is compared within its most selective bucket while keeping the
  O(n) win. EFFECT: recovers merges for popular short titles whose tokens are all common.

[MEDIUM] Series volumes show as separate cards — extract.py:886, catalog.py:644 _group_series,
  catalog_groups.py — norm_title strips a trailing "Vol N" (so "Foo Vol 3/4" collapse — good) but NOT
  mid-title / " - 03" / "(3)" / "第3巻" volume markers, and regroup has NO series handling (it only
  LABELS a group for the View-Series link). So a series ingested as N distinctly-titled volumes shows N
  cards. HOW: extend the volume-strip regex ([-–] 0?\d+$, (\d+)$, CJK 第…巻/卷/권); add a regroup pass
  that clusters members sharing a normalized series key (or external series id) into one card with
  per-volume members. EFFECT: collapses per-volume duplicate cards — biggest over-cardinality source for
  comics/LNs.

[LOW] media_bucket split for partially-detected comics — catalog_groups.py:96, catalog.py:347 _media_bucket —
  media_kind upgrades text→comic only when a page trips the comic signal, so the same work seen as comic
  on one source and text on another lands in two buckets and never merges. HOW: when a K1 identity_key ties
  two rows of different buckets, allow the merge (prefer the comic bucket) / propagate media_kind=comic across
  rows sharing norm_key or identity. EFFECT: fixes the comic-vs-text split.

[LOW] Source picker lists near-identical editions — catalog.py:655 dedupe_sources — dedupes by
  (domain, norm_key) but keeps edition variants, so one domain can contribute several near-identical source
  rows. HOW: dedupe by (domain, edition_core = norm_key minus _EDITION_MARKERS), keeping the richest.
  EFFECT: cleaner source pickers (not a card-count change).

14B — PERSIST MATCHES / CUT REDUNDANT API CALLS
--------------------------------------------------------------------------------

[HIGH] Re-enrichment re-searches by title; matched id is thrown away — catalog_enrichment.py:234-237
  _enrich_provider — best_match() does a fresh search() (1 AniList GraphQL + ranobedb call, sometimes a
  cleaned-title retry) then fetch(ref): 2-3 calls per provider per row EVERY enrich, and bm[1].ref
  (AniList id / ranobedb id) is discarded (only genres/themes persisted). HOW (uses K1): persist
  extra["enrich_ref"]={provider: ref}; on re-enrich, if a ref exists call provider.fetch(ref) directly (1
  cacheable by-id call) and SKIP search(). EFFECT: ~50-66% fewer provider calls on any re-enrich of an
  already-matched row.

[HIGH] No negative cache: transient-fail rows are re-searched every ~90s forever — catalog_enrichment.py:243-263 —
  a row where all providers are unavailable raises _Transient and is NEVER stamped, so the
  `enriched_at IS NULL` selection re-picks it next tick and re-searches AniList+ranobedb+OL indefinitely;
  the only guard is a process-local _domain_cooldown lost on restart. HOW: add extra["enrich_attempts"] +
  extra["enrich_next_at"] (exponential backoff) on transient/empty results; gate the selection query on
  enrich_next_at<=now OR null; cap attempts so a permanently-dead title stops being swept. EFFECT: removes
  the dominant waste for catalogs with many unmatchable web-index rows.

[HIGH] Three paths search the same providers for one work, never sharing — catalog_enrichment._enrich_provider
  vs metadata_sync.match_and_enrich_work vs matchmeta.get_work_meta — catalog enrichment, hooked-Work
  matching (writes MetadataLink.ref), and acquire-time alt-title fetch each independently search AniList/OL
  for the same logical work (a popular title gets searched on AniList 3+ times). HOW: route all three through
  the K1 persisted external-id map: fetch-by-id when a MetadataLink/enrich_ref already exists for the work's
  norm_key; seed the catalog row's enrich_ref from MetadataLink.ref. EFFECT: collapses 3 searches per work
  into 1 search + cheap by-id fetches.

[MEDIUM] One-at-a-time fetch where batch multi-id exists — metadata.py:513 AniListProvider.fetch,
  metadata_sync.check_releases:299-355 — check_releases loops links calling fetch(ref) one GraphQL round-trip
  each (throttled ~1/s). AniList supports Page(media:{id_in:[…]}) ~50/req; Hardcover supports
  book_id:{_in:[…]}. HOW: add fetch_many(refs) using the paged id_in query; batch link refs into pages of 50.
  EFFECT: AniList release-watch calls drop from N to ~N/50 (~98% cut for large linked libraries).

[MEDIUM] No HTTP response cache; in-process caches lost on restart — metadata.py:104-106 _request,
  series.py:32 _SERIES_CACHE, book_catalog _resolve_seen — _request builds a fresh AsyncClient per call with
  no ETag/Cache-Control; by-id fetches (stable, highly cacheable) always hit upstream; _SERIES_CACHE (1h) and
  _resolve_seen (6h) are process-local dicts, so a restart re-fetches all warm data. HOW: (a) wrap provider
  GETs in an on-disk httpx cache (hishel) honoring ETag/Cache-Control so unchanged by-id fetches return 304;
  (b) persist _SERIES_CACHE/_resolve_seen to a DB table/AppSetting keyed by norm_key with the existing TTLs.
  EFFECT: avoids full re-downloads on every poll/restart; meaningful for the 90s enrich tick + 6h sync on hot
  titles.

[MEDIUM] Series enumeration re-runs 5 calls per title after restart/expiry — book_catalog.py:736-767,
  series.py:277-376 — detect_series does Hardcover series-search + series-books + OL filter + OL author + GB
  (~5 calls) per title, memoized only in-process for 1h; series membership is stable but the expensive
  enumeration isn't persisted (only the resulting extra.series annotation is). HOW: persist the resolved
  enumeration under extra["series_members"] (member norm_keys + positions) with a long TTL/version so a repeat
  detect_series re-annotates from the DB without the 5 calls unless stale. EFFECT: repeat View-Series / series-
  acquire after restart goes from 5 calls to 0.

[LOW] Domain cooldown lost on restart re-trips blocks — catalog_enrichment.py:70-89,423 — _domain_cooldown /
  anilist cooldown are in-memory, so a restart re-probes a still-blocked domain (e.g. CF-blocked comix) and eats
  a fresh block. HOW (optional): persist cooldown-until per domain in AppSetting. EFFECT: a few saved calls per
  restart.

NOTE: meta_checked (book_catalog) and match_meta_at (matchmeta) are already write-once permanent timestamps that
correctly gate against re-calls — leave them as-is (they are the DESIRED persist-once behavior); only add a TTL if
upstream staleness ever matters.

SUGGESTED ORDER FOR SECTION 14

1. K1 (persist external id / identity_key + enrich_ref) FIRST — it is the shared foundation for

ROOT CAUSE #1 [DATA] — the catalog popularity signal is ~absent → "Top N" selects junk
--------------------------------------------------------------------------------
  EVIDENCE: every stock item has popularity_norm < 0.01 and 68% are exactly 0.0. The
  "Top 400 books" surfaces obscure public-domain + foreign-language long-tail: "The boke
  of Saint Albans", Dutch "Vanden Vos Reinaerde", German "Die zärtlichen Schwestern",
  Persian "Hamah bā-savād mī shavaym", Indonesian "Media Pembelajaran MI/SD", "Men of the
  Old Stone Age". These are NOT on the 2 enabled usenet book indexers and match libgen
  poorly. This is the dominant cause of low availability — garbage selection in.
  WHY: book catalog popularity is enriched from OpenLibrary readinglog/ratings (Section
  14B) but for these long-tail/foreign rows it is null→0, so the "Top N by popularity"
  ordering degenerates to ~arbitrary, and the catalog is dominated by Gutenberg/OL
  public-domain imports rather than actually-popular titles.
  FIX:
   (a) [DATA] Gate stock selection on a real popularity FLOOR (e.g. require
       popularity_norm above a percentile, or OL readinglog_count/ratings_count > N), and
       EXCLUDE items with popularity 0 / unenriched from auto-stocking. A "Top N" must
       draw from enriched, above-floor rows only — never from the zero-pop tail.
   (b) [DATA] Seed the stock pool from a real popularity source (Hardcover/Google Books
       bestseller/ratings, or NYT/Goodreads-style lists) rather than the crawl's
       public-domain-heavy catalog. Public-domain Gutenberg titles should be stocked via
       libgen/gutenberg DIRECTLY (they're freely available), not via usenet.
   (c) [CODE] In the stock selector, language-gate to the pipeline languages (['en']);
       a German/Dutch/Persian title can NEVER match an en-only usenet/verify path — don't
       queue it for usenet at all (route to a language-appropriate source or skip).

--------------------------------------------------------------------------------
ROOT CAUSE #2 [OPS] — only 2 usenet book indexers enabled → thin coverage
--------------------------------------------------------------------------------
  EVIDENCE: Prowlarr has 16 indexers but only NZBFinder + NzbPlanet are usenet+enabled;
  all others are torrents and disabled. "Dune" → 40 results, but "Twisted Love Ana Huang"
  (a popular romance) → 0, and every obscure title → 0.
  FIX:
   (a) [OPS] Add/enable more usenet book indexers in Prowlarr (e.g. additional NZB
       indexers the operator has access to) — coverage of 2 indexers is the hard ceiling
       on usenet availability.
   (b) [OPS/CODE] Consider enabling torrent protocol for books (config currently
       protocols:['usenet'] only) with a torrent client, OR lean harder on libgen for the
       long tail. Right now usenet-only + 2 indexers + obscure catalog = near-zero hits.

--------------------------------------------------------------------------------
ROOT CAUSE #3 [CODE] — doomed grabs waste SAB + fill the daily cap (false positives)
--------------------------------------------------------------------------------
  EVIDENCE (journal): the matcher grabs releases that then fail verify and cascades to the
  next candidate, each a real SAB addurl:
   • Author-as-title catalog rows: stock "title" is actually an author — "William Morris",
     "Knut Hamsun" → searched as titles → "verify FAILED title 0.00 · author ?" against
     releases where that name is the AUTHOR. Guaranteed-doomed, yet 4-5 candidates grabbed.
   • Same-title-different-book false positives: "Telling Lies" → grabbed "Telling Lies
     Online" / "Telling Lies About Hitler" → title 0.90 · author miss · conf 0.45 → fail.
   • No-author items (72/400): title-only matching can't ever confirm author → every grab
     fails verify, but the cascade still tries up to CANDIDATE_CAP(6) per item.
  RESULT: 1261 grabs → 3 stocked. Floods the shared SAB (memory: shared with Sonarr) and
  burns the libgen/usenet daily cap on hopeless items.
  FIX:
   (a) [CODE] PRE-GRAB verifiability gate (release_matcher / downloads.grab_release): do
       NOT grab a candidate whose pre-computed title_author_confidence is below the VERIFY
       floor it will be judged by — today the stock/fuzz path grabs sub-verify-floor
       releases and lets download+verify reject them. Align the grab floor with
       verify._VERIFY_MIN so we never download a release we already know will fail.
   (b) [CODE] Refuse title-ONLY grabs when the book has no author metadata (the dominant
       false-positive source) — route those to libgen-by-title or mark unavailable instead
       of grabbing 6 doomed usenet candidates.
   (c) [CODE] Detect author-as-title rows: when norm(title) ≈ norm(author) or the title is
       a bare person-name, treat the row as malformed (skip / re-derive title) rather than
       searching the author as a title. Fix the upstream catalog mapping that put the
       author in the title field.
   (d) [CODE] Early-abort the cascade: if the first K candidates all fail verify with
       title < a low bound, stop grabbing the rest (they're the same bad search results).
   (e) [CODE] Lower the per-item grab ceiling for low-confidence/stock items; reserve the
       wide cascade for high-confidence books only. (FUZZ_CANDIDATE_CAP=25 is far too high
       for speculative grabs against a shared SAB.)

--------------------------------------------------------------------------------
ROOT CAUSE #4 [CODE] — libgen fallback matches little for the long tail
--------------------------------------------------------------------------------
  EVIDENCE: for "The Story of Florence" libgen returned 50 hits + annas 50 hits, but
  candidates_for filtered ALL to 0 ("primary providers found nothing"), then zlibrary +
  oceanofpdf browser renders FAILED. Earlier libgen JOBS failed "blocked/unreachable
  (transient)" at the DOWNLOAD/resolve step (not search — search works from this host).
  FIX:
   (a) [CODE] The browser-render fallbacks (zlibrary "page is navigating"; oceanofpdf
       "Target page closed") are effectively dead — fix the render waits (await network
       idle / a stable selector before page.content) or drop them from DEFAULT_PROVIDERS
       so they stop wasting a headless render per item. (Ties to F0.4 / Section 13B browser
       fixes.)
   (b) [CODE] candidates_for is too strict for obscure titles (50 hits → 0 candidates):
       apply the Section 13A fuzzy scorer + E1 CJK norm + language/format ranking so a real
       obscure-but-present edition isn't filtered out; and for libgen prefer fetch-by-MD5
       editions in the work's language.
   (c) [CODE] The libgen "blocked/unreachable (transient)" on download/resolve: instrument
       which host/step returns "throttled" (the resolve mirror is often a different,
       CF-fronted host than the search host) and rotate to the nginx mirrors (.bz/.vg/.li)
       which are NOT Cloudflare-fronted, before giving up. Ties to Section 13B libgen
       mirror health-ranking + Range-resume.
