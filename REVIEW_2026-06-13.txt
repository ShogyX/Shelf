[RISK-1] downloads.py sweep_orphan_staging() climbs to the SHARED drop-zone root.
   _staging_root() = os.path.dirname(_job_dir(storage_path)) — i.e. the SAB
   category "complete" dir. The GC then rmtree's every entry there that no current
   DownloadJob references and that is older than the 2h grace.

     "Deliberately does NOT climb further — climbing to the shared drop-zone root
     could make verification scan and match a file from a DIFFERENT download."
     The new code climbs exactly one level past that boundary, then DELETES there.
   - Memory note: this SAB (10.10.102.27) is SHARED with Sonarr. Isolation holds
     ONLY while Shelf's SAB category ("shelf" by default) has a dedicated complete
     dir. If an operator points Shelf at a shared/parent complete dir (or SAB's

   safety margin is thin and contradicts the sibling function's stated invariant.
   RECOMMEND: gate the sweep to entries whose name matches Shelf's own job-naming,
   or confine to category dir AND assert root basename == _category(integ); at

   VERIFIED: authors_compatible('村田 雄介','尾田 栄一郎') True→False; same author

[FIXED — HIGH] norm_title emptied standalone titles to a BLANK grouping key.

   whole titles: "V 2"→"", "S 1"→"", "C 137"→"". A blank key is refused by the
   union-find guard, so the title was silently stranded (never grouped/searchable).
   FIX: after marker-stripping, if the title would collapse to empty, fall back to
   the pre-strip form. VERIFIED: "V 2"→"v 2", "C 137"→"c 137"; real volume collapse
   intact ("Berserk Vol 1"/"Berserk v2"→"berserk", "Naruto - 7"→"naruto").

[FIXED — MED] work_title_from truncated intrinsic-colon titles ("Re:Zero"→"Re").

   "Re:Zero kara…" preserved whole; "Dune: Part One"→"Dune", "Spider-Man: No Way
   Home"→"Spider-Man" still trim correctly.

[FIXED — HIGH] The new NO_AUTHOR_MIN_CONF=0.9 gate broke the ENTIRE comic pipeline.
   comix_catalog rows carry NO author (only media_kind="comic"), so the author-less

--- DIAGNOSIS ---

     specific challenge needs a newer, Turnstile-capable solver.

--- WHAT WAS BUILT (the requested Cloudflare solver integration) ---
A solver-agnostic FlareSolverr-compatible integration, wired in app-wide:

   * solve(url): POST {cmd:request.get} to the proxy's /v1; returns the solved HTML,
     the earned cookies (cf_clearance/__cf_bm) and the exact UA. Treated as an
     operator-trusted internal service (private IP) — NOT routed through the public
     SSRF guard, exactly like the Prowlarr/SABnzbd clients. Never raises.
   * ensure_clearance(url): solves the SITE ROOT (an HTML page — solving a JSON API
     URL directly makes FlareSolverr time out), caches the cf_clearance + UA per host,
     locked so concurrent crawlers solve at most once, reused for a TTL (~25 min).
   * Failed-solve back-off: a host the solver can't pass is not re-attempted for 10
     min, so a crawl fails FAST instead of paying a ~60s solver timeout per page.
   * clearance_for / invalidate / configured.

[CONFIG] config.py: SHELF_FLARESOLVERR_URL (+ _TIMEOUT_S, _CLEARANCE_TTL_S). Empty =
   disabled (falls back to the in-app renderer). Set in backend/.env (gitignored) to
   http://10.10.102.23:8191 and confirmed loaded by the running app.

[WIRED — comix API] comix_catalog._fetch_page now: plain GET (replaying any cached
   clearance cookie + pinned UA) → if challenged AND a solver is configured, solve the
   root once, then retry the API GET with the clearance. Graceful no-op without a
   solver (same as before).

[WIRED — app-wide HTML] PoliteFetcher.get_html: on a solvable CHALLENGE it now tries
   the external solver FIRST (_solver_retry: earn clearance, inject the cookies into
   the plain-HTTP jar, pin the solver's UA, retry the plain GET), and only falls back
   to the in-app headless render if the solver is absent or can't solve it. So EVERY
   crawl request that hits Cloudflare uses the solver when relevant.

cf_clearance is (IP, UA)-bound, so the design solves ONCE and replays the cookie+UA
over cheap plain HTTP rather than proxying every request (the solver is slow + serial).

[VERIFIED]
   * Live end-to-end against the REAL solver: solve("example.com") → status 200 +

   * 8 unit tests (tests/test_flaresolverr.py): configured-gating, clearance caching +

--- WHAT THE COMIX FAILURE ACTUALLY IS (two stacked defenses) ---
1. CLOUDFLARE TURNSTILE: comix.to returns 403 `cf-mitigated: challenge`. VERIFIED that

   Cursor/refresh/completion semantics preserved; the old token-API path kept as
   `_ingest_tick_api` (+ the FlareSolverr integration from Part 7 still serves other CF

PART 9 — UNIFIED AUTO-ESCALATING CLOUDFLARE LADDER (all crawl requests)
================================================================================
Goal: every request the app makes should automatically cycle through its request +
CF-solve types, so a site that suddenly adds OR escalates a Cloudflare challenge is
handled without code changes. Full suite after this: 667 passed, 4 skipped.

THE LADDER (PoliteFetcher.get_html — the shared path ALL crawl adapters use: web-index
indexer, comix, jnovel, generic_feed, libgen, metadata):
  Tier 0  plain HTTP (replays any cached clearance)
  Tier 1  FlareSolverr  — external solver earns cf_clearance, replayed cheaply over HTTP
  Tier 2  in-app headless render (Playwright)
  Tier 3  zendriver (headful, Xvfb) — passes Turnstile/managed challenges the others can't
Each tier is tried ONLY on a detected challenge, in cost order; after Tier 1/2 the result
is re-checked and escalates further if STILL challenged. The winning tier is remembered
PER HOST (`_host_solver`) + the render budget stays sticky, so the next fetch of a
CF-flipped host goes straight to the tier that works — automatic adaptation.

[NEW] app/ingestion/cf_browser.py — generic standalone zendriver solver:
  `python -m app.ingestion.cf_browser <url>` → {status, html, body_text, cookies, ua}
  after verify_cf. Run as a subprocess (Xvfb) so the headful browser never touches the
  app event loop. (comix_browser stays the specialized /browse pager; shares _chrome_path.)
[NEW] app/ingestion/zendriver_solver.py — async invoker: available() gate (zendriver +
  xvfb-run present), per-host failed-solve cooldown (no repeated slow timeouts), returns
  the solved page or None.
[FETCHER] _zendriver_render builds a RenderedPage from the solve, injects its clearance
  cookies + UA into the plain-HTTP jar (cheap replay after), and marks the host sticky.
  _result_is_challenge uses looks_like_challenge_page (short + structural marker + no
  images) so a SOLVED long/image-rich page that merely still embeds the /challenge-platform/
  script is NOT mis-flagged (would otherwise loop the ladder forever) — unit-tested.
  Forced/sticky renders ALSO escalate to zendriver when still challenged (fixes comix
  chapter fetching, which uses force_render).

[VERIFIED] generic `cf_browser` solve vs comix.to: status 200, 568KB real HTML,
  cf_clearance earned, looks_like_challenge_page=False (real content). 3 escalation unit
  tests (ladder fall-through + stickiness, solved-vs-interstitial detection, no escalation
  on a non-challenge ban). All crawl adapters route through get_html → inherit the ladder.
