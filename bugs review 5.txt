[HIGH] app/ingestion/fetcher.py:739-749
FlareSolverr clearance cookies discarded by set_identity → replay always re-challenges
  _solver_retry injects the earned cf_clearance/__cf_bm cookies into the live client
  (742-743), THEN calls self.set_identity(cl.user_agent, ...) (746). Since the solver's
  UA almost always differs from the app UA, set_identity retires that client
  (_stale_clients.append, _client=None). The following self.get(...) (749) builds a
  fresh, cookieless client → the plain-HTTP replay is challenged again, defeating the
  entire FlareSolverr tier. FIX: adopt the UA first (rebuild client), then inject the
  cookies into the resulting live client before the GET.

--------------------------------------------------------------------------------
[HIGH] app/ingestion/fetcher.py:709-717
zendriver clearance cookies set on a client that set_identity immediately discards
  Same ordering defect in _zendriver_render: clearance cookies set on `client`
  (712-715), then set_identity(data["user_agent"], ...) (717) retires that client when
  the solver UA differs. The host is marked sticky "zendriver" and future "cheap plain
  GETs" rebuild a cookieless client, so the just-earned clearance is lost and every
  subsequent fetch re-solves. FIX: set identity/UA before injecting cookies.

[HIGH] app/ingestion/fetcher.py:534-544
Crawler GET validates host then connects by hostname — DNS-rebinding SSRF gap (main path)
  get() runs assert_public_url(cur) (536) but then client.get(cur, ...) (544) by
  hostname, causing a SECOND DNS resolution at connect time. assert_public_url returns
  the validated IPs precisely so callers can pin the connection (netguard._pin_to_ip /
  safe_get), but the fetcher discards them. A name that resolves public for the check
  and internal at connect (DNS rebinding) bypasses the guard. The render / zendriver /
  capture_canvas paths share the gap (assert then navigate by hostname). FIX: pin to one
  returned IP with Host header + sni_hostname, as safe_get does.

[MEDIUM] app/ingestion/indexer.py:311-315, 414-420  (vs fetcher.py:569-590)
A plain (non-challenge) 403 cools the whole site down
  The fetcher intentionally returns a markerless 403 as ordinary content (fetcher 569-590,
  comment "a members-only/paywalled 403 ... is handled as usual"). But _fetch_one sees
  status>=400 (415) and _classify_status buckets 403 as "blocked" (313), so
  _handle_fetch_failure cools the entire domain with escalating backoff. One forbidden
  URL throttles the whole crawl — opposite of the fetcher's design. FIX: classify a clean
  403 as a per-page skip/permanent outcome in the indexer, not a site-wide block.

--------------------------------------------------------------------------------
[MEDIUM] app/ingestion/fetcher.py:789
Render path records telemetry "success" BEFORE the render is attempted
  _render calls telemetry.record(hostname,"crawl","success") at 789 — before the rate
  acquire, the SSRF-guarded navigation, the retry loop, and the block check. A render
  that times out / retries / raises RateLimited was already counted as success, and the
  failure path records nothing further → the request dashboard over-reports success and
  never reflects render-side blocks. FIX: record the outcome AFTER the render resolves.

--------------------------------------------------------------------------------
[MEDIUM] app/ingestion/fetcher.py:153-163
Rate-limit lock held across the entire inter-request sleep
  acquire() holds self._lock for the whole `await sleep(wait)` (153-162). With per-host
  concurrency 1 this is harmless, but for any budget shared by concurrent callers (same
  rate_key) every waiter blocks on the lock and recomputes ready_at from the previous
  holder's _last_request_ts, making delays stack additively; a long Retry-After cooldown
  also pins the lock for the full window. FIX: compute the wait + record the reservation
  timestamp under the lock, then release before sleeping.

--------------------------------------------------------------------------------
[LOW] app/ingestion/fetcher.py:286-293
set_concurrency swaps the global semaphore, transiently un-capping concurrency
  Replaces self._semaphore with a new asyncio.Semaphore(n) (293) while in-flight fetches
  still hold slots on the OLD object. Until they drain, effective concurrency can be
  (old in-flight + n), exceeding the cap; lowering n isn't enforced against existing
  holders. Short-lived fetches keep this low-severity. FIX: resize via a counter/condition.

[HIGH] app/ingestion/catalog_groups.py:181-186
Regroup watermark misses catalog deletions
  _should_regroup keys the change-watermark on count(id) and max(coalesce(enriched_at,
  updated_at)). If a row is deleted AND another added between two ticks, count is
  unchanged and max may be unchanged → `changed` is False and grouping silently skips a
  rebuild even though membership changed (e.g. blocklist removal + an old re-add). FIX:
  include min(id)/a deletion counter, or hash over (count,max,min,sum-of-ids), or set a
  dedicated dirty flag on any catalog mutation.

  bucket this spikes memory exactly as the comment says it avoids. FIX: iterate the
